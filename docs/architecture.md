# 架构设计文档

## 1. 系统概述

本系统是一个基于**事件驱动架构**的多 Agent 智能运维平台（AIOps），通过四个专业 Agent 的协作，实现从告警检测到故障自愈的全闭环自动化。

## 3. 核心设计决策

### 3.1 为什么选择事件驱动架构？

| 维度 | 事件驱动 | RPC 直调 |
|------|----------|----------|
| 耦合度 | 松耦合，Agent 互不感知 | 紧耦合，需知道对方地址 |
| 扩展性 | 新增 Agent 只需订阅 topic | 需要修改调用方代码 |
| 容错 | 消息持久化，消费失败可重试 | 调用失败需要自建重试 |
| 背压 | Kafka 天然支持 | 需要额外实现 |
| 审计 | 事件天然可追溯 | 需要额外记录 |

### 3.2 为什么选择状态机编排？

Pipeline（线性管道）的局限：
- 无法处理条件分支（RCA 置信度低时重新分析）
- 无法处理失败重试（Heal 执行失败重试 3 次）
- 无法处理人工介入（Change 被拒需等待）

状态机的优势：
- 条件路由：根据上一步结果动态决定下一步
- 检查点恢复：每步完成后持久化，重启可续
- 可视化：状态转换图清晰展示工作流

### 3.3 Agent 间通信模式

```
Orchestrator 模式（本项目采用）：
  中心化编排，Orchestrator 控制流程
  
  优点：流程可控、易于调试、支持事务
  缺点：Orchestrator 是单点

Choreography 模式：
  去中心化，Agent 通过事件自行协作
  
  优点：无单点、更灵活
  缺点：流程难以追踪、调试困难
```

## 4. 数据流详解

### 4.1 一次完整故障处理的数据流

```
1. Prometheus 抓取指标 → Kafka(aiops.alerts)
2. MonitorAgent 消费 → 异常检测 → 生成 AlertEvent
3. Orchestrator 条件检查 → RCA 需要执行
4. RCAAgent 消费 AlertEvent → 知识图谱查询 → 贝叶斯推理 → RCAEvent
5. Orchestrator 条件检查 → confidence >= 0.3 → Heal 需要执行
6. HealAgent 消费 RCAEvent → 匹配 Playbook → dry-run → HealEvent
7. ChangeAgent 消费 HealEvent → 风险评分 → 审批决策 → ChangeEvent
8. 审批通过 → 执行修复 → 验证恢复 → 关闭事件
```

### 4.2 知识图谱数据模型

```
节点 (Nodes):
  (:Service {name, type, tier, namespace})
  (:Pod {name, status, node})
  (:Node {name, ip, capacity})
  (:Metric {name, type})
  (:Alert {name, severity, timestamp})
  (:Change {description, timestamp, author})

关系 (Relationships):
  (Service)-[:DEPENDS_ON]->(Service)    服务依赖
  (Pod)-[:RUNS_ON]->(Node)              部署关系
  (Metric)-[:MONITORS]->(Service)       监控关系
  (Alert)-[:TRIGGERED_BY]->(Metric)     告警触发
  (Alert)-[:AFFECTS]->(Service)         告警影响
  (Change)-[:APPLIED_TO]->(Service)     变更关系
```

## 5. 安全设计

### 5.1 自愈安全护栏

```
请求 → [限流器] → [dry-run] → [爆炸半径检查] → [熔断器] → [审批门控] → 执行
         │           │              │               │            │
       超限拒绝    模拟失败       半径过大        已熔断       未审批
         ↓           ↓              ↓               ↓            ↓
       告警通知    返回错误      升级为L2       暂停自愈     等待人工
```

### 5.2 分级策略

| 级别 | 操作示例 | 审批 | 爆炸半径 |
|------|----------|------|----------|
| L0 | 重启 Pod、限流、扩容 | 自动 | < 5% |
| L1 | 回滚版本、改配置 | oncall 确认 | 5-20% |
| L2 | 数据库操作、全量回滚 | TL 审批 | > 20% |

## 6. 三语言实现对比

| 特性 | Python | Java | Go |
|------|--------|------|-----|
| Agent 抽象 | ABC 抽象类 | abstract class | interface |
| 异步模型 | asyncio | CompletableFuture | goroutine+channel |
| 依赖注入 | 手动/工厂 | Spring IoC | 手动构造 |
| 类型安全 | 弱(运行时) | 强(编译时) | 强(编译时) |
| 并发模型 | 协程(单线程) | 线程池 | goroutine(M:N) |
| 适用场景 | AI/ML 密集 | 企业后端 | 云原生/运维 |
| 部署体积 | ~200MB(含依赖) | ~150MB(JRE) | ~15MB(静态链接) |
