## 一、项目介绍类（10 题）

### Q1: 简单介绍一下你的 AIOps 项目？
**A**: （使用 STAR 法则，见 star-method.md 的 1 分钟版本）

### Q2: 为什么做这个项目？
**A**: 看到腾讯、美团等大厂在大量招聘 AIOps Agent 方向的工程师，运维智能化是行业趋势。传统运维的告警轰炸和人工排障效率太低，多 Agent 协作 + AI 可以大幅提升效率。

### Q3: 你在这个项目中负责什么？
**A**: 整个系统的架构设计和核心实现，包括四个 Agent 的设计开发、事件驱动架构搭建、知识图谱的数据模型设计、时序异常检测算法选型和集成。

### Q4: 项目有多少人参与？
**A**: 核心开发 2-3 人。我负责 Agent 框架和编排器设计，另外有同事负责数据采集层和前端可视化。

### Q5: 项目的用户量/数据量？
**A**: 监控 50+ 微服务，每天处理约 200+ 告警事件。Prometheus 每 15 秒采集一次指标，Kafka 日消息量约 100 万条。

### Q6: 项目上线了吗？效果如何？
**A**: 已上线内部使用。MTTR 降低 87%（40min→5min），告警噪音降低 85%，60%+ 常见故障实现自动修复。

### Q7: 这个项目最大的挑战是什么？
**A**: Agent 间的状态一致性和故障自愈的安全性。一方面要保证多个 Agent 对同一事件的处理不冲突，另一方面要确保自动修复操作不会扩大故障范围。

### Q8: 你是怎么解决的？
**A**: 状态一致性通过 correlation_id 关联 + 编排器串行控制解决。安全性通过分级策略 + 安全护栏链（dry-run、熔断器、爆炸半径评估）保障。

### Q9: 如果重新做，你会改进什么？
**A**: 三个方面：1) 引入 RAG 增强根因分析 2) 用 RL 优化自愈策略选择 3) 支持跨集群联邦

### Q10: 这个项目和市面上的 AIOps 产品有什么区别？
**A**: 市面产品（如 PagerDuty、Datadog）主要做监控告警和可视化，我们在此基础上加了多 Agent 协作的闭环自愈能力，从告警到修复全自动化。

## 二、架构设计类（15 题）

### Q11: 为什么选择事件驱动架构？
**A**: 三个原因：1) 解耦——Agent 互不感知，可独立迭代 2) 可靠——Kafka 持久化，消费失败可重试 3) 可审计——事件天然可追溯

### Q12: 为什么用 Kafka 不用 RabbitMQ？
**A**: Kafka 更适合高吞吐的日志/指标流处理场景：1) 支持持久化和回溯消费 2) 分区并行消费吞吐更高 3) 运维生态（Prometheus exporter）更成熟

### Q13: Agent 之间如何通信？
**A**: 通过 Kafka Topic 异步通信。每个 Agent 订阅输入 Topic，处理后发布到输出 Topic。编排器通过状态对象在 Agent 间传递上下文。

### Q14: 如果一个 Agent 挂了怎么办？
**A**: 1) Kafka 消息持久化，重启后从上次 offset 继续消费 2) 编排器的检查点机制支持断点恢复 3) K8s 自动重启 Pod

### Q15: 如何处理消息顺序性？
**A**: 同一个 incident_id 的事件发到同一个 Kafka partition（用 incident_id 作为 key），保证单个故障的处理事件有序。

### Q16: 系统的 SLA 是多少？
**A**: 系统可用性目标 99.9%。通过 Kafka 集群（3 节点）、Agent 多副本、检查点恢复保障。

### Q17: 如何处理重复事件？
**A**: 两层去重：1) MonitorAgent 用告警指纹（fingerprint）在 5 分钟窗口内去重 2) 消费端用 event_id 做幂等处理

### Q18: 编排器是单点吗？怎么解决？
**A**: 当前是 Orchestrator 模式，有单点风险。解决方案：1) 编排器多实例 + 分布式锁（如 Redis/etcd）2) 或者改为 Choreography 模式

### Q19: 系统怎么扩展？
**A**: 1) Kafka 增加 partition + Consumer 实例 2) Agent 水平扩容（无状态设计）3) Neo4j 读写分离

### Q20: 如何保证数据一致性？
**A**: 最终一致性。Agent 间通过 Kafka 异步通信，用 correlation_id 关联事件，用补偿机制处理失败。

### Q21: 有没有考虑过用微服务拆分各 Agent？
**A**: 考虑过。当前单进程部署便于开发调试，生产环境可以把每个 Agent 拆为独立微服务，通过 Kafka 解耦。架构上已经为此预留——Agent 间只通过事件总线通信。

### Q22: 监控系统如何监控自己？
**A**: 1) 暴露 Prometheus metrics（Agent 处理延迟、成功率、队列深度）2) 设置告警规则（错误率>5%、P95>5s）3) 结构化日志写入 Loki

### Q23: 日志怎么处理的？
**A**: structlog 输出 JSON 格式结构化日志，包含 correlation_id、agent_name、latency 等字段，便于 ELK/Loki 查询和关联分析。

### Q24: 配置管理怎么做的？
**A**: pydantic-settings 管理配置，支持环境变量覆盖和 .env 文件，K8s 部署时通过 ConfigMap/Secret 注入。

### Q25: API 设计遵循什么规范？
**A**: RESTful 风格。POST 触发事件、GET 查询状态。统一错误格式、分页、版本号（/api/v1/）。

## 三、算法深入类（15 题）

### Q26: 3-Sigma 异常检测的原理？
**A**: 假设数据服从正态分布，计算 Z-Score = |x - μ| / σ。Z > 3 即异常（99.7% 的正常数据在 μ±3σ 内）。

### Q27: EWMA 的 α 参数怎么选？
**A**: α 越大对近期数据越敏感（适合快速变化的指标），越小越平滑（适合稳定指标）。通常 0.1-0.3，可以用网格搜索在历史数据上调优。

### Q28: Isolation Forest 的工作原理？
**A**: 随机选特征、随机选切割点，构建多棵隔离树。异常点因为"特殊"更容易被隔离（路径短），正常点需要更多切割。异常分数与平均路径长度成反比。

### Q29: 为什么用投票而不是单一算法？
**A**: 减少误报。3-Sigma 可能在非正态数据上误报，IF 可能在数据少时误报。投票要求多数算法同意，大幅降低 False Positive。

### Q30: 贝叶斯推理具体怎么算的？
**A**: P(根因|症状) = P(症状|根因) × P(根因) / P(症状)。先验 P(根因) 来自历史统计，似然 P(症状|根因) 根据当前证据评估。例如有近期部署时，P(CPU高|部署引起) 的似然从 0.5 提高到 0.9。

### Q31: BFS 图遍历的时间复杂度？
**A**: O(V + E)，V 是节点数，E 是边数。空间复杂度 O(V)（visited 集合 + 队列）。

### Q32: 知识图谱怎么解决数据实时性？
**A**: 事件驱动更新——部署、配置变更等事件触发图谱异步更新。全量同步 + 增量更新结合。

### Q33: 如果根因分析的置信度很低怎么办？
**A**: 编排器有条件路由——置信度 < 0.3 时跳过自愈，直接通知 oncall 人工排查。也可以触发二次分析，收集更多证据。

### Q34: 风险评分模型的权重怎么确定的？
**A**: 初始权重基于经验设定，上线后可以用历史数据做回归分析调优。爆炸半径权重最高（0.3）因为它直接关系到影响范围。

### Q35: Prophet 的优势是什么？
**A**: 自动分解时序为趋势 + 季节性 + 节假日效应。适合有明显周期性的业务指标（如 QPS 的日周期、周周期）。

## 四、编码实现类（20 题）

### Q41: Python 的 asyncio 和 Java 的 CompletableFuture 有什么区别？
**A**: asyncio 是单线程协程（事件循环），适合 I/O 密集。CompletableFuture 基于线程池，适合 CPU 密集。Go 的 goroutine 是 M:N 调度，兼顾两者。

### Q42: 为什么 Python 版用 FastAPI 而不是 Flask？
**A**: FastAPI 原生支持 async/await、自动生成 OpenAPI 文档、类型校验更严格（Pydantic）、性能比 Flask 好。

### Q43: Java 版用了哪些设计模式？
**A**: 模板方法（BaseAgent）、策略（不同 Agent）、工厂（Orchestrator 创建）、观察者（事件总线）、责任链（安全护栏）、状态（熔断器）。

### Q44: Go 版的并发模型怎么设计的？
**A**: 每个 Agent 可以是独立的 goroutine，通过 channel 传递事件。用 sync.Map 存储共享状态，atomic 操作做无锁计数。

### Q45: Go 的 interface 和 Java 的 interface 区别？
**A**: Go 是隐式实现（鸭子类型），只要方法签名匹配就实现了接口。Java 是显式实现（implements 关键字）。Go 更灵活但缺乏编译时强制检查。

## 五、系统设计类（15 题）

### Q61: 如果让你设计一个告警收敛系统，怎么设计？
**A**: 三层收敛：1) 指纹去重（相同告警 5 分钟内只报一次）2) 拓扑聚合（同一服务链路的告警合并）3) 降噪过滤（低置信度告警抑制）

### Q62: 如何设计一个高可用的 Agent 编排系统？
**A**: 1) 编排器多实例 + 分布式锁 2) 检查点持久化到数据库 3) 事件总线（Kafka）保证消息不丢 4) Agent 无状态设计，可水平扩展

### Q63: 如何处理告警风暴（一次性来了 1000 条告警）？
**A**: 1) 告警指纹去重 2) 时间窗口合并 3) 知识图谱拓扑关联（找到根因后抑制下游告警）4) 限流保护

### Q64: 如何设计审批流程？
**A**: L0 低风险自动通过 → L1 oncall 确认（Slack/钉钉通知）→ L2 TL 审批。超时自动升级。所有决策记录到审计日志。

### Q65: 怎么保证自愈操作不会造成更大故障？
**A**: 五层防护：1) dry-run 模拟 2) 爆炸半径评估 3) 熔断器（连续失败停止）4) 分级审批 5) 自动回滚（执行后验证失败则回滚）

## 六、开放问题（10 题）

### Q76: AIOps 的未来趋势？
**A**: 1) LLM Agent 会替代大量规则引擎 2) 多 Agent 协作成为主流 3) RL 优化运维决策 4) 全栈可观测性（Metrics + Logs + Traces 统一）

### Q77: 你怎么看 LLM 在运维中的应用？
**A**: LLM 擅长理解非结构化数据（日志、文档）和生成修复方案，但不能完全替代传统算法（时序检测、图遍历）。最佳实践是 LLM + 传统算法混合。

### Q78: 这个项目有什么技术债务？
**A**: 1) 异常检测算法需要更多真实数据调优 2) 知识图谱的自动更新还不够完善 3) 缺少 A/B 测试框架

### Q79: 如果给你无限资源，你会怎么完善？
**A**: 1) 引入 LLM 做智能日志分析 2) 构建故障预测模型（提前预警）3) 做跨集群全局根因分析 4) 建设完整的 A/B 测试和回归测试平台

### Q80: 你是怎么学习 AIOps 相关技术的？
**A**: 1) 研究开源项目（HolmesGPT, Aurora, Microsoft AIOpsLab）2) 阅读论文和行业博客 3) 动手实现核心算法 4) 关注大厂技术分享

## 面试技巧总结

1. **先总后分**：先说架构全景，再深入某个模块
2. **有数据**：MTTR 降低 87%，误报降低 85%，这些数据要记牢
3. **有深度**：每个技术点都能往下说 2-3 层（如 Kafka → ISR → acks=all）
4. **有取舍**：说清楚为什么选 A 不选 B（如事件驱动 vs RPC）
5. **有反思**：主动说改进点，显示你在持续思考
